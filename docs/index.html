What is Keel?

In shipbuilding, the keel is the first structural member laid down — every other part of the vessel attaches to it. These documents are that keel for a software project: the load-bearing spec that you (working with Claude Code) construct the actual product from.

Keel is a Claude Code skill that interviews you about a greenfield idea through a rotating panel of expert personas, then generates a tailored, cross-referenced suite of foundational documents you can build from.

Instead of asking you to fill in a template or write a brief before you code, Keel shows up as a conversation partner — one that challenges weak assumptions and synthesises your answers into a coherent spec before a single line of code is written.

Keel is optimised for use with Claude Code as the primary build agent. The documentation it generates is structured to serve as a durable source of truth for AI-assisted development.

Why Keel?

Most documentation tools either dump a template on you, or generate docs from a one-line description. Keel does neither.

  • Interview-first Never generates a document from a one-line idea. The interview is where the value is created — by surfacing what you haven't thought through yet.
  • Adaptive scope A weekend prototype and a multi-tenant regulated platform get very different doc sets. Keel proposes only what your project actually needs.
  • One fact, one home No copy-paste between documents. Every fact lives once and is cross-referenced by a stable requirement ID from every document that depends on it.
  • Threat-modeled Optionally hardens the generated spec before you write a line of code. Weaknesses are fixed in the design documents, not just noted in a risk register.
  • Optimised for Claude Code The output is the source of truth for the AI builder. CLAUDE.md is the keystone — every other document hangs off it by stable reference.

When to use Keel

Keel is designed for the very start of a greenfield project — before you've written code, chosen a stack, or committed to a structure. It's also useful when a project has grown organically and you need to reconstruct the foundational spec retroactively.

It is not a substitute for ongoing documentation practices, sprint planning, or production runbooks — though it generates the initial versions of those artefacts too.

Quick Start

Two commands and you're ready to lay the keel of any new project.

Install via plugin marketplace

/plugin marketplace add achatt89/keel
/plugin install keel@thelogicatelier

Restart Claude Code after installation. Then, in any project folder, tell Claude Code:

"I want to start a new project, let's lay the foundations" — or just type /keel

What happens next

Keel will run an intake to understand your idea and set an ambition tier (Prototype, MVP, or Platform), then move through a staged panel of expert personas — each one building on what was said before. At the end, it proposes a doc set, generates everything with cross-references wired in, and optionally runs a threat-modeling pass.

Set aside a focused session — the interview is where the value is created.

Prerequisites

Before installing Keel, make sure the following are available in your environment.

  • Claude Code — installed and working. Verify with claude --version in your terminal.
  • git — on your PATH. Verify with git --version.
  • GitHub access — for plugin marketplace install, you'll need either gh auth login or an SSH key configured for GitHub. This requirement goes away once the repository is public.

Keel has no npm, pip, or runtime dependencies. It is a plain Claude Code skill — a set of Markdown instructions the agent reads and acts on.

Personal Skill Install (Method B) →

Personal Skill Install

Method B — install directly from the repository. A symlink keeps the skill updated when you pull.

User-global install (symlink)

git clone https://github.com/achatt89/keel.git ~/code/keel
mkdir -p ~/.claude/skills
ln -s ~/code/keel/plugins/keel/skills/keel ~/.claude/skills/keel

The symlink approach means a git pull inside ~/code/keel automatically updates the skill — no reinstall needed. Restart Claude Code after cloning.

Update

git -C ~/code/keel pull

Then restart Claude Code.

Copy instead of symlink

cp -R ~/code/keel/plugins/keel/skills/keel ~/.claude/skills/keel

You'll need to re-copy to pick up updates.

Project-scoped install (team use)

To make Keel available only within a specific project — useful for teams where not everyone has it installed globally — symlink or copy into the project's own skills directory: 

mkdir -p .claude/skills
ln -s ~/code/keel/plugins/keel/skills/keel .claude/skills/keel

Commit the .claude/skills/keel directory (not the symlink) if you want the skill to travel with the repo.

Verify & First Run →

Verify & First Run

Verify installation

Open Claude Code and type /keel. If the skill is installed correctly, it should appear as an available command. Alternatively, say:

"I want to start a new project, let's lay the foundations"

Claude Code will route this to Keel automatically.

First run

Open your target project folder (or an empty folder for a new project) in Claude Code, then describe your idea. Keel will run through its full flow:

  1. Intake — reflect the idea back, establish ambition tier
  2. Interview — six staged persona interviews (see The Interview)
  3. Proposed doc set — tailored to your project's complexity
  4. Generation — documents written with cross-references wired in
  5. Threat-model pass (optional)
  6. Handoff — reading order, open decisions, first build step

The interview is where the value is created. A focused 30–60 minute session will produce a far stronger spec than rushing through the questions.

How the interview works →

The Interview

Keel conducts a staged multi-persona interview before writing a single document. Each persona adapts every question to what was just said — it challenges weak assumptions instead of transcribing them.

  • Phase 0 Intake — reflect the idea back, establish ambition tier (Prototype / MVP / Platform), and set expectations for the session ahead.
  • Phase 1 Business Analyst — the real problem, who pays, why now, and what success looks like in concrete metrics.
  • Phase 2 Product Manager — user personas, key journeys, the MVP cut, and what is explicitly out of scope.
  • Phase 3 Architect — stack choice, data model, integrations, and the hard technical bets that will be difficult to reverse later.
  • Phase 4 Security / Compliance — data sensitivity, tenancy model, authentication approach, and regulatory exposure.
  • Phase 5 Designer — surfaces, key user flows, tone, accessibility needs, and brand feel (only if the project has a UI).
  • Phase 6 Delivery / Ops — phasing, deployment environment, git workflow, worktree parallelism, and doc-sync strategy.
Document Tailoring →

Document Tailoring

Keel does not stamp out the same fifteen files every time. It proposes a doc set matched to your project's real complexity.

By ambition tier

  • Weekend prototype — CLAUDE.md + README, possibly a lightweight PRD if the idea needs clarifying before building.
  • Product / MVP — full requirements suite (BRD, PRD), architecture, engineering design, and an implementation plan with concrete exit gates.
  • Platform / regulated — everything above plus NFR, COMPLIANCE, THREAT_MODEL, RUNBOOK, and optionally an almanac/ knowledge base.

Cross-references and stable IDs

Every document is generated with stable requirement IDs (e.g., REQ-001, ADR-003). When a requirement from the BRD drives a design decision in the architecture, the architecture document references the requirement ID — not a paraphrase of its content. This means the doc set stays coherent as individual documents evolve.

One fact, one home. If you update a requirement in the BRD, you know exactly which other documents reference it and need to be reviewed.

Threat Modeling →

Threat Modeling

After generating the doc suite, Keel offers an opt-in threat-modeling pass that hardens the spec before any code is written.

What it models

The threat-modeling pass works adversarially across seven dimensions:

  • Data exposure — what data could leak and to whom
  • Tenant isolation — whether multi-tenant boundaries are enforced at every layer
  • Untrusted input — all attack surfaces that accept external data
  • Secrets and auth — key management, token lifetimes, privilege escalation paths
  • Abuse and DoS — rate limiting, resource exhaustion, quota enforcement
  • Privacy and regulatory — PII handling, consent flows, retention limits
  • Availability — single points of failure, disaster recovery, RTO/RPO

How it mitigates

Weaknesses identified during the threat-modeling pass are fixed directly in the design documents — not just annotated in a separate risk register. The architecture document and engineering design are updated in place, and each threat is recorded in THREAT_MODEL.md with a pointer to where it was mitigated.

Handoff →

Handoff

After generation (and optional threat-model), Keel delivers a structured handoff so you know exactly where to start building.

What the handoff covers

  • What was generated — a list of every document created, with a one-line summary of each
  • Recommended reading order — the sequence in which a new collaborator (human or AI) should read the docs to build context efficiently
  • Open decisions — all [NEEDS DECISION] markers surfaced from across the doc set, so nothing is forgotten
  • First build step — a concrete starting point, typically: "open this folder in Claude Code and start at IMPLEMENTATION_PLAN.md Phase 0"

The recommended first action after a Keel session: open your project folder in Claude Code, point it at CLAUDE.md, and say "let's start building."

Document Catalog →

The Full Catalog

Every document Keel can generate, and the question it answers.

Document What it answers
CLAUDE.md Keystone index — invariants, document map, working agreements, and current project status. Every other document is referenced from here.
BRD.md Business requirements — the problem being solved, objectives, success metrics, constraints, and risks.
PRD.md Product requirements — user personas, key journeys, feature list, scope cuts, and roadmap.
ENGINEERING_DESIGN.md Design pillars, data classification policy, and non-negotiable technical constraints.
ARCHITECTURE.md System components, data flows, trust boundaries, security controls, and accepted risks.
HLD.md / LLD.md High- and low-level design — sequence flows, module map, API interfaces, and key types.
ADR.md Architecture Decision Records — every significant technical bet, with context, consequences, and revisit triggers.
NFR.md Non-functional requirements — performance, security, cost, and scalability targets, each with a concrete verification method.
DESIGN.md UX and UI design system — tokens, component patterns, accessibility requirements, and brand feel. Generated only if the project has a UI.
COMPLIANCE.md Data protection obligations, retention schedules, consent requirements, and lawful basis for processing. Generated only if the project holds sensitive data.
THREAT_MODEL.md Threat register with mitigations wired into the architecture. Each threat points to where it was fixed. Generated only if the project holds sensitive data or was threat-modeled.
IMPLEMENTATION_PLAN.md Phased build plan with concrete exit gates — the sequence in which Claude Code (or a human team) should build the system.
COMMANDS.md How to run the project — dev server, build, test, migrate, seed, and any environment-specific variations.
RUNBOOK.md How to operate and recover the system — deployment steps, monitoring, alerting, and incident response.
almanac/ Optional topology-stable knowledge base — positioning, methodology, go-to-market strategy, AI training data. Useful for products with strong editorial or brand dimensions.
Contributing →

Contributing

Keel is open source. Contributions — whether new templates, interview persona improvements, or bug fixes — are welcome.

Plugin structure

plugins/keel/ ├── .claude-plugin/ │ └── plugin.json — plugin manifest (name, version, author) └── skills/keel/ ├── SKILL.md — the skill definition (the agent's brain) └── references/ ├── personas.md — interview persona definitions ├── catalog.md — document catalog and selection rules ├── conventions.md — ID schemes, cross-reference syntax └── templates/ — one Markdown template per document type

Adding a document template

  1. Create a Markdown file in skills/keel/references/templates/.
  2. Follow the conventions in conventions.md — stable IDs, status markers, and cross-reference syntax.
  3. Register the document in catalog.md with a clear description of when it should be generated.
  4. If the document introduces a new persona question, add it to the relevant phase in personas.md.

Pull requests

  • Branch off main.
  • Keep changes focused — one concern per PR.
  • Update CHANGELOG.md with a brief entry under the appropriate version.
